# 视觉 AI 模块

## 你的任务

为文章生成两类视觉素材的提示词：
- 封面图：3 组差异化创意
- 内文配图：3-6 张，按段落匹配

无论用户有没有生图 API，你都必须输出完整的提示词交付物。默认将结果保存到该篇文章目录下的 `generated/image-prompts.md`。如果没有 API，这个文件本身就是最终交付物。

---

## 图位是一体化产物

- 图位不是“后面再说”的补充项，而是正文结构的一部分
- `article.md`、`generated/image-prompts.md`、本地图片文件必须一一对应
- 封面固定目标路径为 `assets/cover-wide.jpg` 和 `assets/cover-square.jpg`
- 内文配图固定目标路径为 `assets/img-01.jpg`、`assets/img-02.jpg`、`assets/img-03.jpg`……
- 每个图项都必须同时给出：
  - 槽位编号
  - 目标文件名
  - 插入位置
  - alt 文案
  - 图例说明
  - 对应提示词
- 正文必须插入对应 Markdown 图片引用，例如：

```markdown
![配图 1：这里是 alt 文案](img-01.jpg)
*图 1：这里用一句话解释这张图在说明什么。*
```

- 正文图片和图例说明必须连续相邻：图片引用行下一行直接写图例，二者之间禁止空行。
- 无论是否有生图 API，都先创建同名占位图文件
- 如果后续拿到真实图片，直接覆盖同名文件，不改 Markdown 路径
- 这三件事必须视为一个整体：
  - 正文里的图位
  - 目录里的图片文件
  - `generated/image-prompts.md` 里的提示词和说明

## 总原则

### 账号视觉定位：烧 Token 的人

本项目默认服务账号「烧 Token 的人」。视觉不是泛科技封面，也不是抽象 AI 海报，而是“AI 大模型工程师把复杂问题摊开讲清楚”的视觉语言。

优先使用这几类视觉母题：

| 内容类型 | 首选视觉母题 | 适合主题 | 默认主题倾向 |
|----------|--------------|----------|--------------|
| 模型新闻 / 产品发布 | token 账单、调用链路、产品界面、发布说明拆解 | 新模型、新 API、新产品能力 | `token-clean` |
| AI 科普 / 工程解释 | 调试台、日志窗口、流程图、提示词与输出对照 | RAG、Agent、上下文、推理成本 | `token-mint` |
| 产业链 / 商业分析 | 数据新闻卡、供应链地图、成本结构、公司关系图 | GPU、云厂商、模型公司、应用层 | `token-data` |
| 工具体验 / 工作流 | 真实工作场景、屏幕界面、任务流、前后对比 | AI 工具、效率流、产品测评 | `token-clean` 或 `token-mint` |

视觉关键词：冷白、石墨黑、青绿色强调、少量琥珀色警示、数据新闻、工程日志、API 账单、调用链路、产品截图感、GitHub release、论文页面、控制台、成本卡片、产业链图谱。

不要默认使用这些套路：霓虹赛博城市、发光芯片、蓝紫渐变、机器人握手、抽象大脑、悬浮 3D 字母、廉价科技光效、股票 K 线冲天、金色财富感、土黄色背景、过度金融投研风。

所有封面和内文图都要回答一个问题：这张图有没有让读者更快理解“这个 AI 事件、模型能力、产品体验或 token 成本到底意味着什么”。如果只是让画面看起来更“AI”，就不要放。

### 0. 审美底线：不是 PPT 背景加大字

现在默认按 image2 / gpt-image-2 这类强生图模型来写提示词，不能再把提示词写成“单调背景 + 居中大标题 + 几个装饰图标”的 PPT 封面。每张图都必须像一张经过视觉总监 brief 的完整作品，而不是临时占位图。

每张图必须先定义清楚一套具体视觉方案：

- **具体主体**：画面里必须有一个可被描述、可被看见的主体，例如真实场景、物件装置、人物动作、业务现场、仪表盘、数据卡组、隐喻装置，而不是只有渐变背景和标题。
- **视觉类型**：明确选择一种图像语言：纪实摄影感、编辑部商业摄影、杂志级信息图、3D 静物装置、电影感场景、手绘/矢量插画、拼贴海报、产品海报、数据新闻版式等；不能只写“科技感”“高级感”。
- **构图层次**：必须写清前景 / 中景 / 背景、主体占比、视角、留白、安全覆字区、读者第一眼动线。非信息图的视觉主体建议占画面 55%-80%，文字区不应成为唯一主体。
- **审美细节**：必须写清材质、光线、色彩关系、景深、纹理、空间纵深、边缘质感、阴影、反射或颗粒感等至少 4 类可执行细节。
- **文案与画面关系**：标题和标签要嵌入版式，服务画面，不要像 PPT 模板一样把一行大字贴在空背景中央。封面可以有标题，但标题不能替代画面主体。
- **完成度目标**：提示词里要明确要求“像可直接发布的中文商业媒体视觉 / 杂志头图 / 数据新闻插图”，让人看到会觉得这张图是认真设计过的。

以下都判为不合格提示词，必须重写，不能调用 API：

- 纯色或渐变背景 + 巨大标题
- 抽象光效背景 + 几个漂浮图标
- 空白背景 + 数据大字报
- 只有 PPT 模板感的卡片堆叠，没有真实视觉主体
- 只写“高级、科技、简洁、现代”，没有主体、镜头、光线、材质和构图细节
- 为了省事把所有图都写成同一种深蓝背景信息卡

### 0.1 生图额度保护：先预检，再单张生成

生图 API 额度是成本，不允许用“多试几次”弥补提示词粗糙。默认流程必须是：

1. 先写完整提示词。
2. 逐张做提示词可执行性预检。
3. 只在预检通过后调用生图 API。
4. 单张生成、单张验收，通过一张再生成下一张。
5. 只重跑失败槽位，不重跑已经合格的图片。

**禁止批量盲跑**：不要一次性把封面和所有内文图都丢给 API，然后生成完才检查。

**API 调用必须使用本文件里的提示词**：调用 image2 / OpenAI-compatible 图片 API 时，必须使用 `generated/image-prompts.md` 中对应槽位的正向提示词和负向提示词。禁止临时另写 prompt 直接调用 API。

**提示词变更必须先更新文件**：如果某张图生成失败或质量不合格，先在 `generated/image-prompts.md` 里更新该槽位的提示词、负向约束和验收标准，再用更新后的提示词重跑。API 实际调用内容必须能回溯到提示词文件。

**槽位必须一致**：调用 API 前检查槽位编号、目标文件名、正文图位、alt 文案、图例说明和提示词是否一一对应；不得用一个槽位的提示词覆盖另一个槽位的图片。

**同一槽位最多连续重试 2 次**：第 1 次失败必须先分析失败原因并重写提示词；第 2 次仍失败，保留当前最优结果和完整提示词，向用户说明风险，除非用户明确要求继续重试。

### 0.2 提示词可执行性预检清单

每个槽位在调用 API 前，必须逐项通过：

- 画面类型明确：封面 / 信息图 / 时间线 / 对比卡 / 场景图，不允许含糊。
- 主体明确：读者第一眼应该看到什么，主体占画面多少比例。
- 构图明确：比例、主体位置、标题区、数据区、留白区分别在哪里。
- 审美方向明确：视觉类型、参考气质、材质、光线、镜头 / 视角、空间层次都写清楚。
- 元素数量明确：几张卡片、几条时间线、几组标签、几行数据。
- 文字清单明确：主标题、副标题、标签、角标、来源小字逐条列出。
- 数据清单明确：数值、单位、来源、对应位置逐条列出。
- 失败反例明确：明确禁止 PPT 模板感、单调渐变背景、巨大居中文字、抽象背景、纯渐变壁纸、空白图、无卡片、无数据、乱码、错字、错误数字。
- 手机端可读：小字不能承担核心信息；关键数字和标签必须足够大。

任意一项缺失，先重写提示词，不得调用 API。

### 0.3 信息图 / 数据图反失败模板

凡是价格卡、对比图、时间线、榜单、数据图，正向提示词必须包含类似下面的硬约束：

```text
注意：这不是抽象背景图，不是渐变壁纸，不是装饰图，必须是一张完成度高、可直接发布的信息图。
画面主体占据 80% 以上面积，必须出现 {具体主体，如“两张并排报价卡”}。
必须清楚写出以下文字和数据：{逐条列出}。
禁止只生成背景，禁止 PPT 式大字封面，禁止空白画面，禁止没有卡片，禁止没有数据，禁止错误数字，禁止乱码，禁止密集小字。
```

### 1. 提示词必须是中文

- 所有正向提示词、负向提示词、版式说明、数据说明，都用中文写
- 输出的是可直接投喂给中文生图工具的完整提示词，不是几个松散关键词
- 句子要完整，信息要具体，要让另一个模型一眼知道“这张图要长什么样、写什么字、放什么数据”
- 中文表达必须符合中文使用习惯，像真实中文媒体会写的话；禁止把英文提示词硬翻成别扭中文，禁止出现“机器翻译腔”“中式英文回译腔”

### 2. 数据必须具体，不能空泛

如果图片里涉及数据、图表、榜单、对比、时间线、结论卡片，必须明确写出：
- 具体数值
- 单位
- 时间
- 对比关系
- 来源名
- 图表类型
- 每个数据项的大致排版位置

禁止写：
- “一些数据”
- “若干图表”
- “几组对比”
- “显示增长趋势”

要写成：
- “左侧柱状图显示 2024 年 58%，2025 年 73%，右上角角标写‘同比 +15pct’”
- “底部时间轴写 2023、2024、2025、2026，节点由浅蓝到深蓝渐变”
- “中心是两张并排报价卡；左卡标题 V4-Flash，三行价格分别是 0.028、0.14、0.28 美元；右卡标题 V4-Pro，三行价格分别是 0.145、1.74、3.48 美元”

### 3. 画面必须一眼知道在说什么，不能乱堆元素

- 每张图都要有一个非常明确的“主信息”，读者第一眼就知道这张图在解释什么
- 元素服务表达，不服务热闹；图里每个图标、人物、卡片、箭头都必须对应正文里的一个信息点
- 禁止为了“显得丰富”胡乱堆砌人物、屏幕、图表、装饰物、漂浮图标、无关背景元素
- 如果一句话说不清这张图的核心用途，说明提示词还没写到位
- 能用一张主卡 + 两三组辅助信息讲清楚，就不要硬塞成“四宫格 + 时间线 + 柱状图 + 场景图”的拼盘
- 尤其是数据图，必须让读者看完知道结论，而不是只觉得“信息很多”

### 4. 中文必须清晰可读，绝不能乱码

只要图中出现文字，必须在提示词中明确要求：
- 仅使用简体中文
- 中文必须准确、完整、清晰可读
- 禁止乱码、错别字、异体字、繁简混用、无意义符号、拼音代替汉字
- 禁止把中文渲染成随机笔画、方框字、伪汉字

建议直接写进提示词的硬约束：

```text
画面中的所有文字仅使用简体中文，中文必须自然、准确、清晰可读，不能出现乱码、错别字、异体字、拼音替代汉字、繁简混用、无意义符号。
```

### 5. 文字不要贪多，但出现时必须“少而准”

- 封面图：优先 1 组主标题 + 1 组副标题/角标
- 信息图：优先 3-8 组关键信息标签
- 避免满屏密密麻麻小字
- 宁可少写几组字，也不要牺牲可读性
- 如果一张图需要超过 8 组文字或 6 个关键数字，优先拆成多张图，或改为正文卡片/表格；不要把复杂表格直接交给生图模型赌准确率。

### 6. 如果工具的中文能力不稳，要同时给兜底方案

除了主提示词，再额外给出：
- 后期覆字说明
- 需要手工覆盖的中文文案清单
- 安全留白区域说明

注意：兜底方案是补充，不是替代。主方案仍然要明确要求“中文直出且不能乱码”。

### 6.1 image2 生成后的验收标准

每张 image2 生成图都必须人工查看后再继续下一张，验收项：

- 是否和槽位用途一致。
- 是否出现提示词要求的主体，而不是抽象背景或空图。
- 关键中文是否大体可读，不能出现明显乱码或错字作为核心信息。
- 关键数字是否出现，且没有明显错位或错误。
- 画面是否适合手机端阅读，核心内容不能太小。
- 图例说明是否仍然匹配生成结果。

验收失败时，只重写该槽位提示词并重跑该槽位；不得重跑全部图片。

### 7. 只能使用文章里已经确认过的真实信息

- 只使用 Step 3b 已确认的真实素材和 Step 5 已确认的最终稿内容
- 不得为了好看补造数字、来源、排名、机构名
- 如果正文没有足够数据，就做场景图或概念图，不强行造图表

## 一、封面图

### 双封面硬规则

- 每篇文章必须同时交付两张封面
- 横版封面：`cover-wide.jpg`
- 方形封面：`cover-square.jpg`
- 横版比例固定为 **2.35:1**
- 方形比例固定为 **1:1**
- 两张图必须共用同一视觉主题、同一主标题逻辑、同一色调体系
- 方形封面不能只是横版封面的机械裁切，必须重新适配主体位置、文字层级和留白结构
- 默认发布到公众号草稿箱时使用 `cover-wide.jpg`
- `cover-square.jpg` 作为该篇文章的配套封面物料，保存在文章目录中

### 创意策略

每次至少给 3 组创意，且必须差异明显：

**创意 A：token 账单型**
- 用“账单、调用次数、成本曲线、上下文窗口、缓存命中、推理链路”等具体对象表达文章核心判断
- 适合模型新闻、API 价格、Agent 成本、产品效率、工程实践
- 画面要像一张被工程师认真标注过的成本解释图，而不是金融广告

**创意 B：工程现场型**
- 用调试台、控制台、产品界面、GitHub 页面、论文页、会议白板、工作流看板承接问题意识
- 适合技术科普、工具体验、产品拆解、工作流复盘
- 画面可以有生活感，但不要摆拍成“精英程序员海报”

**创意 C：数据新闻型**
- 用简洁的信息图、公司关系、供应链地图、成本结构、时间线或对比卡传递关键信息
- 适合测评、清单、盘点、复盘、行业分析、产业链观察
- 涉及投资相关内容时，只呈现产业逻辑、成本结构和竞争格局，不做股价暗示，不画买卖点

### 封面图输出格式

每组创意必须输出以下字段：

```markdown
### 封面创意 A: {创意名称}
- 横版槽位编号：cover-wide
- 横版目标文件名：assets/cover-wide.jpg
- 横版比例：2.35:1
- 方形槽位编号：cover-square
- 方形目标文件名：assets/cover-square.jpg
- 方形比例：1:1
- Markdown 引用：封面不插入正文，但发布时默认使用 `{article_dir}/assets/cover-wide.jpg`
- 视觉策略：{token 账单型 / 工程现场型 / 数据新闻型}
- 适合原因：{为什么这篇文章适合这条创意}
- 画面核心：{一句话说清这张图最重要的视觉中心}
- 视觉类型：{纪实摄影感 / 编辑部商业摄影 / 杂志级信息图 / 3D 静物装置 / 电影感场景 / 矢量插画 / 拼贴海报等}
- 主体与审美钩子：{具体主体是什么，为什么它比“背景加大字”更值得看}
- 必须出现的中文：
  - 主标题："{中文标题}"
  - 副标题/角标："{如果需要}"
- 必须出现的数据：
  - {数据1：数值、单位、时间、来源、位置}
  - {数据2：数值、单位、时间、来源、位置}
- 横版构图：{2.35:1 横版、主体位置、留白位置、信息区位置}
- 方形构图：{1:1 方形、主体位置、留白位置、信息区位置}
- 画面层次：{前景 / 中景 / 背景分别有什么，主体占比，标题区不超过多少}
- 版式：{主标题放哪里，数据卡放哪里，图标放哪里；横版和方形分别说明}
- 色调与材质：{主色、辅色、材质、纹理、光影、景深、空间纵深、风格}
- 中文排版要求：{字体气质、字重、字号层级、行数控制}
- 审美验收点：{生成后用什么标准判断它像认真设计过，而不是 PPT 占位图}
- 横版正向提示词：
  "{一整段超详细中文提示词，必须包含 2.35:1 构图、具体主体、视觉类型、前中背景、材质、纹理、光线、镜头/视角、中文文字要求、数据内容、版式约束、反 PPT 约束}"
- 方形正向提示词：
  "{一整段超详细中文提示词，必须包含 1:1 构图、主体重排、留白、安全文字区、材质、光线、中文文字要求、数据内容、版式约束、反 PPT 约束}"
- 负向提示词：
  "{明确写出不要出现的内容，尤其是 PPT 模板感、单调渐变背景、巨大居中文字、空背景、乱码、错别字、密集小字、低清晰度、杂乱背景}"
- 后期覆字说明：{如果直出中文风险高，说明后期该怎么覆盖}
```

### 封面图写法要求

- 横版必须指定 **2.35:1**
- 方形必须指定 **1:1**
- 如果文章是数据型内容，封面应至少带 1-3 个最关键数字
- 中文标题尽量控制在 4-12 个字内，副标题尽量控制在 6-18 个字内
- 标题和数据卡必须留出清晰可读空间
- 封面不是论文摘要，不要把全部结论塞进去
- 如果使用图表，优先柱状图、折线图、数字卡片、时间轴，不要复杂三维图
- 账号名「烧 Token 的人」可以作为小角标出现，但不要抢主标题
- 封面里出现 `token` 时，优先保留英文小写 `token`；标题里可使用 `Token`，但不要全篇混乱大小写
- 封面底色优先冷白、近白、浅灰、石墨黑；暗色图可以用，但不要默认深蓝、蓝紫、霓虹赛博
- AI 产业链内容可以出现公司、芯片、云、模型、应用层关系，但不要做成“股票涨跌图”或“财富密码封面”

## 二、内文配图

### 先决定该不该配图

写作完成后，逐个 H2 判断是否需要配图：

| 适合配图 | 不适合配图 |
|----------|------------|
| 有真实数据、对比、排名、时间线 | 纯观点输出且篇幅很短 |
| 有场景还原价值 | 紧接上一张图，间距不足 300 字 |
| 有转折、高点、案例节点 | 结尾 CTA 段落 |
| 单段过长，需要调节阅读节奏 | 已有引用块或代码块，视觉信息足够 |

### 配图位置约束

- 总数 3-6 张
- 1500 字左右配 3 张
- 2000 字左右配 4 张
- 2500 字左右配 5-6 张
- 相邻两张图至少间隔 300 字
- 不要在文章第一段之前放图
- 配图插在对应段落之后，不是之前

每张图必须输出以下字段：

### 内文配图写法要求

- 尺寸统一为 **16:9 横版**
- 风格和封面必须同一套色调体系
- 优先做“读手机也看得懂”的图
- 优先做“值得被夸一句图做得好”的图：要有具体场景、主体、质感和构图，而不是只做背景板。
- 内文图之间要有同一视觉体系，但不能每张都长成“深蓝背景 + 大字 + 数据卡”；同一篇文章里至少混合 2 种视觉类型，例如信息图 + 场景图 / 摄影感静物 + 数据新闻图。
- 数据图优先用简单结构，不要塞满复杂坐标轴和细小注释
- 重要数据最多突出 3-6 个点
- 如果正文是对比关系，图里也要把对比关系做出来，而不是只有装饰元素
- 生成提示词后，必须把对应的 Markdown 图位实际插进正文，而不是只在提示词文件里描述
- 每张图插入正文时，必须在图片下方紧跟一行图例说明，例如：`*图 1：中美模型周调用量对比，突出中国模型近四周持续领先。*`
- 图片引用行和图例说明行必须连续相邻，中间不要空行；正确写法是 `![配图 1：...](img-01.jpg)` 下一行直接接 `*图 1：...*`。
- 对应占位图必须落到文章目录，保证预览时已经能看到版面中的图位

### 「烧 Token 的人」内文图位库

写 AI、大模型、产品和产业链时，优先从下面的图位类型里选，不要每次都重新发明视觉语言：

| 图位类型 | 适合放在哪里 | 画面核心 | 注意事项 |
|----------|--------------|----------|----------|
| token 成本卡 | 解释价格、调用次数、上下文、缓存、推理成本时 | 一张主成本卡 + 2-3 个辅助指标 | 只放真实确认数据；没有数据就改成概念图 |
| 调用链路图 | 讲 Agent、RAG、工作流、模型路由时 | 用户请求到工具/API/模型/输出的链路 | 箭头少而清楚，最多 5-7 个节点 |
| 产品界面拆解 | 讲 AI 产品体验、功能更新时 | 一个产品界面 + 3 个标注点 | 不伪造真实 UI，可用抽象界面但不能冒充截图 |
| 来源栈图 | 讲热点溯源、论文、GitHub、官方公告时 | X/Reddit/GitHub/论文/官方来源的分层卡片 | 只写已确认来源名，不补造链接和机构 |
| 产业链地图 | 讲芯片、云、模型、应用、数据时 | 上游到下游的关系图 | 不出现买入/卖出/目标价暗示 |
| 真假进展对照 | 讲 demo、产品、工程化、商业化差异时 | 左右对照卡或四阶段进度条 | 文案要短，像编辑判断，不像 PPT 培训页 |
| 工程现场静物 | 文章连续信息密度太高时 | 控制台、笔记本、账单、咖啡、标注纸张 | 用来调节节奏，不要替代关键信息图 |

默认优先级：有真实数据先做信息图；有流程就做链路图；有产品体验就做界面拆解；只有观点时才做工程现场或视觉隐喻。
