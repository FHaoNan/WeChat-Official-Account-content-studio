# 排版升级手册

这份手册只负责一件事：让 WeWrite 产出的公众号文章不再只是「标题 + 连续 H2 + 大段正文」，而是有层次、有节奏、有视觉记忆点的成稿。

对「烧 Token 的人」来说，排版不是装饰。排版的作用是把 AI、模型、产品和产业链里的复杂信息拆开，让读者在手机上能快速抓住：

- 发生了什么。
- 这事和我有什么关系。
- 它到底省 token，还是更会烧 token。
- 哪些是事实，哪些是判断，哪些还要等真实数据。

本文件只使用当前 converter 已支持的结构：`:::callout`、`:::timeline`、`:::dialogue`、`:::quote`、Markdown 表格、图片 + 图例。不要发明新的容器语法。

## 核心原则

1. 手机端优先。第一屏要让读者看见核心问题或核心判断，而不是一段背景介绍。
2. 排版服务信息分层。事实、判断、争议、成本、时间线、对比、图解，要用不同结构承载。
3. 每篇文章至少使用 2 种非纯正文模块。优先用 `callout`、`dialogue`、`timeline`、`quote`、表格、图位。
4. 列表可以用，但不要把核心信息全塞进 `ul/ol/li`。公众号里长列表会显得很散。
5. 图片不是最后补丁。正文图位、图例说明、`generated/image-prompts.md` 必须一一对应。
6. 模块不能为了花哨硬塞。每个模块都要解决一个阅读问题：提重点、降复杂度、做对比、放证据、给判断。

## 先选版式方向

写作前先为这篇文章选定一个「版式方向」和一个「排版主题」。如果 `style.yaml` 里 `theme_mode: auto`，系统会通过 `scripts/layout_strategy.py` 自动生成 `{article_dir}/generated/layout-plan.json`，里面包含 `theme`、`layout_family`、`layout_variant`、`module_pattern` 和 `suggested_module_pattern`。

写作时优先参考 `suggested_module_pattern`，但不要机械照抄。文章结构永远优先于模板。

| 版式方向 | 适合内容 | 推荐主题 | 注意 |
| --- | --- | --- | --- |
| 专业信息流 | AI 产业链、公司动态、商业判断 | `token-data`、`token-clean`、`bytedance`、`github`、`professional-clean` | 信息密度高，但要用表格和 callout 降噪 |
| 科技杂志感 | 模型新闻、产品趋势、工具体验、工程科普 | `token-clean`、`token-mint`、`tech-modern`、`midnight`、`bauhaus`、`github` | 适合结构图、链路图、参数/价格对比 |
| 编辑部评论感 | 观点文、争议判断、行业观察 | `token-clean`、`token-mint`、`sspai`、`warm-editorial`、`newspaper` | 适合 quote、dialogue、短段落 |
| 聚焦型强观点 | 快评、风险提醒、反常识选题 | `focus-red`、`bold-navy`、`bold-green` | 少用，避免每篇都像警报 |
| 克制留白感 | 慢科普、长期趋势、方法论 | `minimal`、`minimal-gold`、`ink` | 适合少图、强段落节奏 |

### 主题选择规则

- 用户明确指定主题时，尊重用户。
- `theme_mode: auto` 时，`style.yaml` 里的 `theme` 只是兜底，不是锁死。
- `theme_mode: fixed` 时，才视为固定主题。
- 最近 2 篇用过的同一主题家族，尽量避开。
- AI 内容不要默认蓝紫赛博风。科技感可以来自结构、留白、数据和产品截图，不一定靠渐变。
- 拿不准时，正文定稿后运行 `toolkit/cli.py gallery` 做主题画廊预览，再决定最终主题。

## 文章开头

开头不要平铺背景。前 4 段内必须出现核心问题、冲突或判断。

### 可用开头

**1. 账单开头**

适合：模型价格、Agent、API、工具调用、产品成本。

```text
这类新闻我一般不先看发布会。

我先看价格页。

因为模型到底有没有变强，很多时候不是 benchmark 告诉你的，是账单告诉你的。
```

**2. 场景开头**

适合：产品体验、工具测评、工程实践。

```text
一个 Agent 最尴尬的时刻，不是答错。

是它答错了，还继续调用工具，继续烧 token，继续表现得很自信。
```

**3. 冲突开头**

适合：热点解读、争议、反常识判断。

```text
这次更新看起来是模型能力升级。

但我更愿意把它看成一次成本结构调整。
```

**4. 对话开头**

适合：科普、概念解释、读者容易误解的选题。

```markdown
:::dialogue
这是不是又一个更聪明的模型？
> 不一定。先看它把哪部分成本挪走了。
:::
```

### 不要这样开头

- 「最近大家都在关注……」
- 「随着人工智能的发展……」
- 「事情的起因是……」
- 「本文将从三个方面分析……」
- 「在 AI 时代，某某变得越来越重要……」

## 专用模块库

下面这些「专用模块」不是新语法，而是用现有容器组合出来的写法模板。

### 1. 先看结论卡

用途：第一屏或 H2 开头，快速告诉读者这篇文章要解决什么问题。

语法：`:::callout info`

```markdown
:::callout info
先看结论：这次更新真正值得看的不是模型名字，而是它把多轮调用里的成本压力往下压了一截。但这只在高频任务里有意义，普通闲聊用户感知不会太强。
:::
```

使用规则：

- 每篇最多 2 个。
- 内容必须是判断，不是摘要。
- 不要写成「本文认为」。

### 2. token 成本卡

用途：把价格、上下文、缓存、输出 token、多轮调用等成本问题从正文里抬出来。

语法：`:::callout warning` 或表格。

```markdown
:::callout warning
token 账单最容易失控的地方，不是用户问的那一句，而是 Agent 在后台反复规划、检索、调用工具、读取结果，再重新规划。
:::
```

适合放：

- API 价格变化。
- Agent 多轮调用。
- 上下文变长带来的隐藏成本。
- 缓存、路由、压缩、检索对成本的影响。

不要放：

- 空泛感慨。
- 和成本无关的品牌口号。

### 3. 工程链路拆解

用途：把一个 AI 能力拆成输入、处理、调用、输出、成本点。

语法：`:::timeline`

```markdown
:::timeline
**输入** 用户提出一个任务，但任务本身往往不够精确。
**规划** 模型先拆步骤，这一步已经开始消耗 token。
**调用工具** 检索、浏览器、代码执行、数据库查询都会带来额外延迟。
**读结果** 工具返回内容越长，后续上下文越贵。
**再规划** Agent 如果没做对，会重新走一遍链路。
:::
```

使用规则：

- 适合解释 Agent、RAG、工具调用、多模态、推理模型。
- 每个节点尽量一句话，不写成长段。
- 节点里要出现真实成本点或风险点。

### 4. 真假进展对照表

用途：防止文章被发布会话术带跑，区分 demo、产品、工程化、商业化。

语法：Markdown 表格。

```markdown
| 看起来像 | 真正要验证的是 |
| --- | --- |
| 模型更聪明 | 真实任务成功率有没有提升 |
| 上下文更长 | 长上下文里检索和定位是否稳定 |
| Agent 会干活 | 工具调用失败后能不能恢复 |
| 价格下降 | 多轮任务总账单有没有下降 |
```

使用规则：

- 表格不要超过 5 行。
- 每格尽量短，手机端可读。
- 适合放在中段，不要一上来就铺大表。

### 5. 来源栈

用途：让读者知道文章不是凭空判断，但又不把正文写成参考文献。

语法：`:::callout info`

```markdown
:::callout info
这篇主要看三类来源：官方价格页、GitHub issue 里的开发者反馈、以及 Reddit / Hacker News 上普通用户对延迟和稳定性的讨论。社交平台只当线索，关键事实以官方信息为准。
:::
```

使用规则：

- 适合热点解读和产业链文章。
- 不需要每篇都放。
- 如果信息源有冲突，用这个模块说明「哪些确定，哪些还不确定」。

### 6. 争议问答

用途：把读者脑子里的质疑提前写出来。

语法：`:::dialogue`

```markdown
:::dialogue
那是不是模型越便宜，大家就越敢用了？
> 不一定。单次调用便宜，不代表完整任务便宜。Agent 真正花钱的地方经常在后台循环。
:::
```

使用规则：

- 一组 `dialogue` 控制在 2-4 轮。
- 问题要像读者真的会问的，不要像客服 FAQ。
- 回答要短，后面正文再展开。

### 7. 截图金句

用途：放一句能被读者带走的判断。

语法：`:::quote`

```markdown
:::quote
模型能力看 benchmark，产品能力看留存，工程能力看账单。
:::
```

使用规则：

- 每篇 1 个就够。
- 不要为了金句牺牲准确性。
- 避免宏大口号，优先写具体判断。

### 8. 图位 + 图例

用途：把复杂结构变成一张图，而不是继续堆文字。

语法：Markdown 图片 + 下一行图例。

```markdown
![配图 1：Agent 多轮调用的 token 成本链路](img-01.jpg)
*图 1：一次看似简单的任务，真正烧 token 的地方往往藏在规划、工具调用和结果回读里。*
```

使用规则：

- 图片引用行和图例行必须连续，中间不要空行。
- 图例解释「为什么这张图放在这里」，不是重复 alt。
- 结构图、对比图、价格卡、时间线图优先；少用抽象科技背景图。

## 四类 AI 文章骨架

### 骨架 A：模型新闻解读

适合：新模型、新 API、新价格、新 benchmark、新开源模型。

推荐结构：

1. 账单或冲突开头。
2. `:::callout info` 放核心判断。
3. H2：这次到底变了什么。
4. 对比表：能力、价格、上下文、速度、适合任务。
5. `:::callout warning` 放误区：不要只看发布会或 benchmark。
6. 图位：价格卡 / 能力对照图 / 调用链路图。
7. H2：普通用户、产品团队、开发者分别受什么影响。
8. `:::quote` 收束一个工程判断。

最低模块：`callout` + 表格 + 图位。

### 骨架 B：AI 技术科普

适合：RAG、Agent、推理模型、上下文、embedding、MCP、多模态、微调。

推荐结构：

1. 一个真实问题开头。
2. `:::dialogue` 拆掉最常见误解。
3. H2：先用人话解释概念。
4. `:::timeline` 拆工程链路。
5. H2：它解决什么，不解决什么。
6. `:::callout warning` 放边界和坑。
7. 图位：结构图 / 流程图 / 对比图。
8. 结尾给一个普通读者能带走的判断。

最低模块：`dialogue` + `timeline` + 图位。

### 骨架 C：AI 产品体验

适合：ChatGPT、Claude、Gemini、Cursor、Perplexity、NotebookLM、各类 Agent 产品和效率工具。

推荐结构：

1. 真实任务场景开头。
2. H2：我让它完成什么任务。
3. `:::callout info` 放一句体验结论。
4. 表格：好用、不稳定、会烧 token 的地方。
5. `:::dialogue` 模拟普通用户最关心的问题。
6. 图位：产品界面关键路径 / 工作流示意。
7. H2：谁适合用，谁不适合。
8. 结尾不要写「值得尝试」这种废话，要给具体使用边界。

最低模块：`callout` + 表格 + `dialogue`。

### 骨架 D：AI 产业链观察

适合：芯片、云厂商、模型公司、应用层、数据、能源、开发工具、资本开支。

推荐结构：

1. 一组钱流或供需冲突开头。
2. `:::callout info` 放产业判断。
3. H2：钱从哪里来，成本流向哪里。
4. 表格：环节、收入来源、成本压力、关键指标。
5. `:::timeline` 放公司或行业动作时间线。
6. 图位：产业链钱流图 / 成本结构图。
7. `:::callout warning` 放不确定性：周期、供需、监管、客户集中度。
8. 结尾说明这是产业观察，不是投资建议。

最低模块：`callout` + 表格 + 图位。

## H2 内部节奏

每个 H2 不要都写成「观点 -> 解释 -> 例子 -> 总结」。

可以轮换这些内部结构：

| 结构 | 适合场景 | 示例 |
| --- | --- | --- |
| 结论先行 | 热点、快评 | 短判断 -> 事实 -> 为什么 |
| 问题拆解 | 科普 | 读者问题 -> 人话解释 -> 边界 |
| 链路展开 | 工程实践 | 输入 -> 处理 -> 输出 -> 成本点 |
| 对比推进 | 产品体验、模型对比 | A 怎么做 -> B 怎么做 -> 差异在哪里 |
| 反面假设 | 观点文 | 如果不是这样呢 -> 哪些事实支持/反驳 |

最低要求：

- 不能连续 3 个大段纯正文没有任何模块。
- 每个 H2 至少有一个节奏变化：短句段、callout、dialogue、表格、图位、quote 任选其一。
- 长段落之后要接短段落，给手机端读者喘气。
- 表格不要超过 5 行，timeline 不要超过 6 节点，dialogue 不要超过 4 轮。

## 图文一体化规则

图位要跟着叙事走，放在读者「刚好需要看到图」的位置。

### 适合放图的位置

- 解释一个链路：放结构图。
- 比较两个模型或产品：放对比卡。
- 讲价格、token、成本：放账单卡或成本流向图。
- 讲产业链：放钱流图或供需结构图。
- 讲时间演进：放时间线图。
- 讲产品体验：放关键界面或工作流图。

### 不适合放图的位置

- 只是为了打断文字。
- 正文已经用表格讲清楚，图片只重复表格。
- 选题需要真实产品界面，但生成一张抽象科技图糊弄过去。
- 每张图都长得像蓝紫渐变科技背景。

### 图例规则

- 每张图下面必须紧跟一小行图例。
- 图例必须解释这张图和正文判断的关系。
- `generated/image-prompts.md` 里的用途、alt、图例说明，要和正文图位逐一对上。
- 封面和正文图保持同一视觉语气，但不要每张图都是同一构图。

## 微信端兼容提醒

- 微信编辑器对复杂布局、外部样式、长表格支持有限，优先使用简洁块状结构。
- 表格要短，列数控制在 2-4 列。
- `:::callout` 内不要放复杂表格。
- `:::timeline` 节点保持短句，避免单节点超过 60 字。
- `:::dialogue` 不要写成长访谈。
- 图片、图例、正文之间不要空太多行。
- 如果去掉颜色，文章层次仍然应该清楚。

## 自检清单

写完后逐项检查：

- [ ] 开头 4 段内已经亮出核心问题、冲突或判断。
- [ ] 全文至少使用 2 种非纯正文模块。
- [ ] 每个 H2 至少有一个节奏变化。
- [ ] 没有连续 3 个大段纯正文。
- [ ] 模块承担了明确功能：结论、成本、链路、争议、对比、证据或图解。
- [ ] 没有发明 converter 不支持的新语法。
- [ ] 表格不超过 5 行，`timeline` 不超过 6 节点，`dialogue` 不超过 4 轮。
- [ ] 图位、图例、`image-prompts.md`、占位图文件一一对应。
- [ ] 主题匹配选题气质，没有默认沿用上篇风格。
- [ ] 如果去掉颜色，正文层次仍然清楚。
